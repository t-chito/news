# Read Later 棚卸し

紙名: `Read Later`
補助ラベル: `今週の棚卸し`

## 調査依頼

Feedly の Saved for Later（あとで読む）に溜まった記事の本文を実際に読み、テーマごとに「これらの記事が結局何を言っているか」を統合したブリーフィングを書く。狙いは、全件を読まなくてもテーマ単位で要点が掴め、かつ自分が今どんな関心に寄っているかが見える棚卸しにすること。記事を 1 件ずつ並べて一言添えるリストにはしない。件数を圧縮して中身に変える。

## 素材の取得

`feedly-saved` skill の `fetch` 操作で Saved for Later の全件を取得する。各記事に `entryId`・`title`・`url`・`origin`・`summary`・`saved_at` が付く。

## 発行するかどうかの判定（重要）

ブリーフィングはテーマに記事が固まっているほど価値が出る。少数の記事がバラバラに保存されているだけのときに発行すると、結局 1 記事 1 テーマのリストに崩れて意味がない。

そこで、取得した保存記事の件数が **30 件未満なら、今回は発行しない**。ファイルを書かず、commit もせず、unsave もしない。「保存が N 件で閾値 30 件に満たないため今回は見送る」と報告して終える。この閾値は、まだ溜まっていない時期に無理にまとめないための下限。

30 件以上ある場合のみ、以降の手順に進む。

## 本文の取得

各記事の `url` から本文を取得して読む。ブリーフィングは本文の中身に基づいて書く。タイトルや要約だけからの推測で書かない。

- `zenn.dev` の記事は、本文がページ HTML の `__NEXT_DATA__` 内に入っている。`curl -sL <url> -A "Mozilla/5.0"` で取得し、`<script id="__NEXT_DATA__">` の JSON をパースして `props.pageProps.article.bodyHtml` を読む（WebFetch では本文が取れないので、この方法を使う）。
- それ以外のサイトは WebFetch で本文を取る。
- スライド共有（SpeakerDeck など）や、本文を持たないページは本文取得を諦める。その記事はブリーフィングの主張の根拠には使わない。
- Feedly の `summary` は、本文がどうしても取れないときの最後の代替に留める。summary だけを根拠に全文を読んだかのように書かない。

## グルーピングとブリーフィングの書き方

- 記事の中身から関心のテーマを読み取って自分でグルーピングする。Feedly のカテゴリ（多くが一律）には引きずられない。テーマ名は内容に即して付ける。
- テーマは無理に細かく割らない。1 テーマあたり複数記事が束ねられる粒度にする。記事数が少なく浮くものは「その他」にまとめる。
- 各テーマについて、記事を順番に紹介するのではなく、テーマ全体として何が言えるかを統合した文章を書く。含めるもの:
  - そのテーマの記事群が共通して示している結論や論点
  - 要点を 2〜4 個の箇条書きで
  - 記事間で食い違う主張や、毛色の違う外れ値があれば指摘する
  - そのテーマで全文を読む価値が高い記事を 1〜3 本名指しし、なぜ読む価値があるか（一次情報・具体性・他に無い切り口など、記事の性質）を一言添える。これは読書を置き換えず、どれを読むか決める手がかりにするため。
- 動機を捏造しない。「なぜ保存したのか」「何を確かめたかったのか」は記録に無い情報なので書かない。書くのは記事が事実として述べている内容と、それを束ねた示唆だけ。

## 冒頭のまとめ

今週の保存全体がどんな関心の形をしているかを 2〜3 行で書く。どのテーマに偏っていたか、新しく出てきた関心、逆に拾えていない視点。これが今の関心の地図になる。

## 関心プロフィールの更新（profile.md）

棚卸しの蓄積を毎回作り直さず育てるために、`readlater-weekly/profile.md` を 1 枚の常緑ドキュメントとして手入れする（Digital Garden の発想）。日付ログを積むのではなく、その場で書き換えて成長させる。

1. `readlater-weekly/profile.md` が存在すれば読む。無ければ新規に作る。
2. 今回のブリーフィングを踏まえて改訂する。
   - 続いている関心は、変化（深まった・分岐した・別テーマと繋がった）を反映して書き換える。
   - 今回初めて出てきた関心は新しい項目として加える。
   - 関連するテーマ同士はリンクや言及で繋ぐ。
   - 既に関心が薄れたと見える項目は、消すのではなく「以前の関心」として位置づけ直してよい。
3. 文書の体裁は「現時点での関心・知識領域の地図」。時系列の議事録にしない。
4. 書き換えた `profile.md` を保存する。これは index.html と一緒にコミットされる。

## 出力構造

`news-publish` の流し込み手順に従う。HTML 断片は次の形にする。

```html
<section class="summary">
  <h2 class="summary__title">まとめ</h2>
  <p>今週の関心の形を 2〜3 行で。</p>
</section>

<section class="topic">
  <h2 class="topic__title">テーマ名</h2>
  <article class="article">
    <div class="article__body">
      <p>テーマ全体を統合した文章。</p>
      <ul>
        <li>要点</li>
      </ul>
      <p class="article__pick-label">全文を読む価値がある記事</p>
      <ul>
        <li><a href="URL">タイトル</a>：理由</li>
      </ul>
    </div>
    <ul class="article__sources">
      <li><a href="URL">出典タイトル</a></li>
    </ul>
  </article>
</section>
<!-- テーマの数だけ section を並べる -->
```

各テーマの `article__sources` には、そのテーマに束ねた記事の URL を列挙する。

## 発行後の処理

push が成功した後に、今回ブリーフィングに含めた記事の `entryId` 一覧を `feedly-saved` skill の `unsave` 操作に渡して Saved for Later から外す。push が失敗した場合は unsave しない。
