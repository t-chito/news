AIコーディングエージェントの運用設計

この束を貫いているのは、エージェントに作らせること自体はもう難問ではなく、運用に乗せる前後の「絞り込み」「検証」「権限の制御」をどう設計するかが勝負だ、という共通認識である。1日平均15.8件のPRを1人で量産した開発、1か月でニュースキュレーターを育てた事例、株・仮想通貨botを300個生成した記録はいずれも、生成の速さよりも、生成物のうち本当に使えるものをどう選び残すかに最も時間と工夫を割いている。bot生成の記事は300個のうち実運用に乗せたのは5個だと明かし、AIが付けた「おすすめ度」は信用せず、別モデル（GPT系）による批判的監査と人力バックテストを必須に据える。「デバッグはもう人間の仕事ではなくなった」も同じ構図を一般化し、5体のdebuggerと1体のcriticを対立させて仮説を相互検証させることで、根拠の薄い結論が上がってこないようにしている。生成と検証を別の役に分ける設計が、複数の独立した筆者から繰り返し提示されている点が、このテーマの中心的な結論である。

もう一つの太い論点が、エージェントに知識と権限をどう与えるかである。「オレたち流のコードを書かせる」「AGENTS.md自動更新」「認知負荷を下げるSkill」「独自タイミングのhooks」は、ルールやドキュメントやワークフローをスキルやhookに切り出して継続更新する手法を扱う。「オレたち流」の記事は「Claudeがすでに知っていることは書かない（一般的ベストプラクティスは省き、チーム固有の流儀だけを抽出する）」という方針を明確に立て、AGENTS.mdの記事も同じ思想で「指示は少ないほど良い、重要なものを上に書く」を引いている。一方で「デバッグはもう人間の仕事ではなくなった」は権限について踏み込み、Claude Codeのpermission機能やPreToolUse hookによるコマンドのブラックリストは原理的に抜け穴だらけだと断じ、クラウド側のIAMとサービスアカウント分離、sandbox-execによる認証トークン隔離で守るべきだと主張する。これはhookやsettings.jsonで安全を担保できると説く一般的な紹介と真っ向から食い違う外れ値であり、注意して読むべき対立点である。

毛色の違いも目立つ。AI-DLCの記事はAWS公式のワークフローを題材に、巨大なルールファイルを一括で読ませる方式（main）と、15個のスキルへ分業させてオーケストレータがルーティングする方式（v2）を、ドキュメント読解レベルで比較し、どちらも模索中だと結論づける。これは「ルールで縛るかスキルで分業させるか」という前述の手法選択を一段上から俯瞰する視点を与える。さらにモデル別の検証記事2本は運用論ではなく実測で、追加耐性テストの記事はモデルが高いほど指示外の「アドリブ」が増えるという直感に反する傾向を示し、レビュー特性の記事はOpus 4.7が読み手志向で辛口、GPT-5.5が原則を文字通り厳守するというレビュー人格の差を720ケースの統計で明らかにする。後者のSafety軸でGPT-5.5が許可リスト外のファイル1個を理由に85点も引く挙動は、「別モデルに批判させる」運用の根拠を具体的に裏づける。

ブラウザ自動化系は、エージェントにPlaywrightやChrome拡張を介して画面操作をさせる実装に寄っている。E2E自動修復の記事は夜間E2Eの失敗を朝までにDraft PRで直させる仕組みで、QA知見をinstructionsに蓄積する自己改善の発想が15.8件PRの記事のlearned-rulesと響き合う。ここで明確な外れ値となるのが「社内Chromeで3連敗した話」で、「全自動化させるより、ユーザーがコピペした方が速く確実な箇所は人に残すハイブリッド設計の方が、社内環境では完走率が高い」と結論する。完全自動化を志向する事例に対し、この記事は社内SSLインスペクションやChromeのプロセス統合挙動という現実の壁を踏んで、あえて人を残す判断に着地している点で、テーマ内の方向性が割れている。

トークン経済とコンテキスト効率

3本に共通するのは、トークン節約を課金対策にとどめず、Claude に渡すコンテキストの設計そのものを問い直す動きとして捉えている点である。Zenn の2本はいずれも Claude Max x20 プランでも使用枠が苦しくなった経験から出発し、消費の主因を会話本文ではなく、設定・ルール・MCP のオーバーヘッド、長く伸びたセッション、大量ファイルの探索的読み込みに見ている。Qiita の1本は問題を Markdown の全文読み込みに絞り込み、ローカル完結で見出しチャンクだけを返す mdq という Skill を作って検証している。狙う層は異なるが、解決の方向は「必要十分な情報だけを与える」という一点に収束する。

具体策は層ごとに整理できる。入出力の層では、CLI 出力を圧縮する RTK や、Markdown を全文ではなくチャンクで返す mdq、出力日本語そのものを削る「原始人」的圧縮が挙がる。記憶の層では、セッション間メモリ（claude-mem、kizami）やローカル LLM による RAG 化で、毎回のコンテキスト再構築を避ける。ワークフローの層では、Skill やフックで探索手順を強制し、手当たり次第のファイル読み込みを排除する。「$40/日から1週間でも余裕」の記事はこの3層を独立導入可能かつ相乗効果ありとして体系化しており、3本のなかで最も俯瞰的である。

AIに知識を渡す設計（LLM Wiki・ナレッジグラフ・RAG・ログ）

これらの記事は、扱う道具がObsidian、Notion、ナレッジグラフ、RAG、CloudWatch Logsとばらばらでありながら、ひとつの同じ主張に収束している。AIの賢さはモデルではなく「渡す知識の作り方」で決まる、という主張である。Pineconeの「モデルを良くしてもエージェントは賢くならない」という言葉、ナレッジグラフ本が引くMITの「生成AIパイロットの95%がROIを生まない、主因は技術ではなく文脈をAIが記憶せず毎回ゼロから始まることだ」という指摘、Karpathyの「質問のたびに知識をゼロから再発見している」という表現は、表現こそ違えど同じ問題を指している。複数の筆者が共通して、生のドキュメントをクエリ時に毎回解釈させるのをやめ、取り込み時に一度だけ知識を構造化して蓄積する方向へ進んでいる。

LLM Wiki系の記事は、Karpathyが2026年4月に公開したGistを起点に、Obsidian VaultをAIが読み書きする知識ベースとして使う実践を報告している。三層構造（人間が置く不変のraw層、AIが生成・保守するWiki層、人間が定義するschema層）と、ingest・query・lintの三操作が共通の骨格である。「繋げる力」の筆者は1ヶ月運用の結果、LLM Wikiの真価が要約ではなく複数ソースを横断した概念ページの自動生成、すなわち「繋げる力」にあると結論づけ、別々に取り込んだ論文群からLLM-as-Judge評価バイアスという一つのパターンが立ち上がった具体例を挙げている。PageIndexはドキュメントを意味的なツリーに変換しLLMが要約を手がかりに探索する手法で、ナレッジグラフ本はベクトル検索が苦手とする関係性の推論を構造で保証する。アプローチは違うが、いずれも「類似性ではなく構造・関連性で検索する」「LLMに型と出典と構造を持たせて渡す」という同じ原則に立つ。

整合性の維持と運用も共通の論点である。「唯一の設計思想」とナレッジグラフ本はともに、LLMが生成したWikiやRAGに誤りが紛れ込み、それが後続の参照で「正しい知識」として定着・汚染する危険を指摘する。これに対し、複数の記事が役割分担で応える。Wiki系はlintをLLMに任せる一方、contextlintはドキュメント間の整合性検証を「AIが書く、静的解析がチェックする」と明確に分離し、再現性が求められるYes/No判定をLLMの確率的出力に委ねない。Notion設計は、目的・背景・議事録・タスク・作業ログ・成果物をリレーションでつなぎ「探す」から「たどる」状態を作る点で、Obsidianのwikilinkやナレッジグラフのエッジと発想を共有する。ログ設計はAIへ渡す知識の話ではなく、コストと用途で保存先を使い分けるAWS運用論であり、テーマの中では外れ値に近い。共通項は「何を残し、いつ捨てるかを設計しないと後で困る」という運用思想の一点にとどまる。

食い違いも見える。Obsidianの「LLM Wiki構築」記事はKarpathy方式の自動増殖がコンテキスト腐敗を招くとして、書くタイミングと整理を人間が握る手動運用へ組み替えた。一方「繋げる力」は自動運用のlintで整合性を保つ立場を取り、両者は「AIにどこまで任せるか」で対立する。「ファンシーHTML化」はAIが書きAI同士でレビューする時代にはMarkdownよりHTMLが読みやすいと主張するが、これはレビュー表示の話で、知識をAIに渡す設計の本筋からはやや離れる。さらに「唯一の設計思想」は、ドキュメントが10万トークン以下なら長文脈モデルに全文を渡す方がRAGやWikiより速くて安い場合があると述べ、こうした仕組みを組む前にまず長文脈で足りるか試す価値を説いている。

  • 共通の核心は「モデルではなく知識の渡し方を変える」こと。クエリ時に毎回生ドキュメントを解釈させず、取り込み時に一度だけ構造化して蓄積し、型・出典・構造を持たせて渡す。
  • LLM Wikiの価値は要約集ではなく、複数ソースを横断した概念ページが自動で組み上がる「繋げる力」と、AIが飽きずにこなすbookkeeping（相互参照更新・矛盾記録・一貫性維持）の肩代わりにある。
  • LLM生成の知識ベースは誤りが定着・汚染するリスクを持つ。lintをLLMに任せる流派と、整合性検証を静的解析へ分離する流派（contextlint）が並立する。
  • 「AIにどこまで任せるか」で意見が割れる。自動増殖を手放し人間が整理を握る運用と、自動lintで保つ運用が対立。規模が小さければ長文脈モデルへの全文投入で十分という反対意見もある。

AI時代のテスト戦略

6本の記事は、テストを「正しさの確認」から「品質を構造として設計し、変化に耐える資産にする」営みへと位置づけ直している点で一本の線につながる。最上流では、狩野モデルや脅威モデリングで当たり前品質を抽出し、ビジネス価値との因果をたどってどこをテストするか絞り込むテストアーキテクチャ戦略が語られる。そこから一段下りると、各機能が単体で正しくてもユーザーの操作フローで不具合が起きるという問題意識のもと、ユーザーの行動を起点にゴールまでの流れを確認するシナリオテストが、単体・結合・システムテストを置き換えずに補う層として置かれる。両者は、機能起点の検証だけでは利用時品質を担保できないという共通認識で噛み合っている。

戦術のレベルでは、振る舞い（何をするか）に焦点を当ててテストを書くことでリファクタリング耐性を高め、その副産物としてテストコードが日本語の「生きた仕様書」になるという主張が核になる。この仕様書は人間だけでなくAIエディタにコンテキストとして与える際にも有効だと述べられ、テストとAIの結合点が示される。AIとの結合点はもう一方からも現れる。QAエンジニアが、改修の影響範囲を「上から辿る」のではなくDBカラム・翻訳キー・トグル・権限・設定値という5つの起点から「逆引き」する探索軸をAIへの指示書として設計した事例である。深いネストに強く取りこぼしのない探索軸を与えるという発想は、シナリオテストが狙う影響範囲の網羅と同じ目的を、AI活用という別の手段で追っている。

足元の現実への接地も二方向から補強される。テストがないレガシーコードに対しては、デバッグプリントで入出力を採取してまず雑なテストで固め、カバレッジを段階的に上げながらリファクタリングへ進む育て方が、年単位の道筋として提示される。境界値については、カバレッジ100%でも比較演算子の境界が検証されていない穴を突いて、コードの境界値と既存テスト値をASTで照合し不足を警告するGoの静的解析ツールが紹介される。前者が「テストの量と信頼を育てる」のに対し、後者は「網羅したつもりの穴を機械的に暴く」点で補完関係にある。

食い違いと外れ値も明確にある。振る舞いテストの記事は、実装詳細に寄せたテストを脆いアンチパターンとして退ける一方、Infrastructure層だけは発行されるSQLそのものが仕様だとして実装（HOW）に寄せたテストを例外的に推奨しており、「振る舞いを見よ」という原則に自ら例外を設けている。レガシーコードの育て方の記事は逆に、管理しづらいと一般に避けられる共有フィクスチャを「この段階では構わない」と一時的に許容し、理想形を初手から求めない立場を取る。整理の枠組みも統一されていない。狩野モデルやBSCといった重量級フレームワークでマクロから攻める戦略記事に対し、QAの記事は「画面UIは5つしかない」という極端に絞った分類で軽量に攻めており、抽象度の置き方が対照的である。技術スタックの面では境界値ツールと振る舞いテストの記事がGoを土台にする一方、レガシー育成の記事はJavaScriptを例にしており、適用領域の広さに差がある点も外れ値として読める。

  • テストは正しさ確認ではなく品質の設計対象であり、上流の品質定義から下流の境界値・レガシー対応まで連続した戦略として束ねられる。
  • 機能起点の検証だけでは利用時品質を取りこぼすという問題意識が、シナリオテストと影響範囲の逆引きという別々の手段で共有されている。
  • 振る舞いベースのテストは「生きた仕様書」となり、人間だけでなくAIへのコンテキストや探索の指示書としても機能するという、AI時代ならではの結合点が複数記事に現れる。
  • 「振る舞いを見よ」の原則にInfrastructure層という例外があり、共有フィクスチャの許容や枠組みの抽象度でも記事間に意図的な食い違いがある。
